## 1. 计划目标

本计划用于指导 TradeMate V1 的研发启动和版本交付，目标如下：

1. 先落地多 Agent 共用底座。
2. 在该底座上交付广告 Agent 首版。
3. 保证插件是主入口，Web 是管理后台。

## 2. 计划假设

### 2.1 研发组织假设

建议最小团队：

1. 产品经理 1 人
2. UI/UX 设计 1 人
3. 前端工程师 2 人
4. 后端工程师 2-3 人
5. QA 1-2 人
6. 技术负责人 1 人

### 2.2 周期假设

建议 V1 首版研发周期为 12 周，不含大规模商用前的持续迭代。

## 4. 12 周排期建议

### 4.1 第 1-2 周：规格冻结与基础工程

目标：

1. 冻结对象模型和 API 契约
2. 确认插件和 Web 页面结构
3. 建立代码仓、工程模板、环境配置

交付物：

1. 需求冻结版本
2. 技术方案冻结版本
3. 插件/Web/后端基础工程
4. 数据库初始化方案

### 4.2 第 3-4 周：平台底座一期

目标：

1. 完成登录、权限、店铺上下文
2. 完成目标配置模块
3. 完成 suggestion / approval / task 基础模型

交付物：

1. 认证与权限 API
2. 目标配置 API
3. 任务状态机基础能力
4. 插件登录态与主框架

### 4.3 第 5-6 周：广告 Agent 建议流

目标：

1. 接通广告数据接入
2. 完成建议生成规则第一版
3. 完成风险分级和待审批流

交付物：

1. 建议生成服务
2. 风险规则配置
3. 插件建议列表
4. Web 待审批页第一版

### 4.4 第 7-8 周：执行与通知

目标：

1. 完成 7 类正式动作的执行器
2. 完成任务中心和通知中心
3. 打通插件和 Web 状态同步

交付物：

1. 执行动作服务
2. 任务中心 API 和页面
3. WebSocket 事件推送
4. 插件提醒和最近任务页

### 4.5 第 9-10 周：复盘与审计

目标：

1. 完成复盘快照生成
2. 完成审计日志
3. 完成目标达成和执行效果展示

交付物：

1. 复盘服务
2. 审计日志 API 和页面
3. Web 复盘分析页

### 4.6 第 11 周：联调与稳定性修正

目标：

1. 全链路联调
2. 修复阻塞性问题
3. 完成性能和异常路径优化

交付物：

1. 联调问题清单
2. P0、P1 缺陷修复
3. 发布候选版本

### 4.7 第 12 周：灰度与发布准备

目标：

1. 完成灰度发布前检查
2. 进行种子用户试用
3. 准备正式发布

交付物：

1. 灰度版本
2. 操作手册
3. 上线检查清单

## 6. 依赖关系

### 6.1 强依赖

1. API 契约冻结后才能稳定推进前后端并行。
2. 目标配置与角色权限完成后，建议和审批才能联调。
3. 任务状态机完成后，执行器和复盘才能稳定接入。

### 6.2 风险依赖

1. 广告数据接入质量决定建议引擎效果。
2. OpenClaw 集成验证结果决定后续执行层扩展路线。
3. 插件主入口交互质量决定首版产品感知。

## 8. 里程碑门禁

### 8.1 M1：规格冻结

通过条件：

1. 功能规格冻结
2. 数据模型冻结
3. 页面结构冻结

### 8.2 M2：底座可用

通过条件：

1. 登录、权限、目标配置、suggestion / approval / task 可跑通

### 8.3 M3：广告 Agent 闭环可用

通过条件：

1. 建议生成、审批、执行、结果回传可跑通

### 8.4 M4：发布候选版本

通过条件：

1. 主要缺陷关闭
2. 验收指标达标
3. 灰度准备完成

## 9. 风险与应对

|风险|影响|应对|
|---|---|---|
|需求继续发散|周期失控|V1 严格限制在广告 Agent|
|数据接入不稳定|建议不可信|先跑通基础数据，再逐步提升规则复杂度|
|执行器不稳定|任务失败率高|动作逐个灰度、幂等控制、失败重试|
|插件入口不顺手|用户不形成使用习惯|先做高频待办入口，不堆过多模块|
|底座抽象过度|研发效率下降|只抽象 V1 和 V2 确定会复用的部分|

## 10. 发布建议

1. 插件采用受限灰度发布。
2. Web 后台先开放给内部和种子用户。
3. 高风险动作默认保持人工审批。
4. 首月重点观察建议采纳率、任务成功率、留存和失败原因分布。
