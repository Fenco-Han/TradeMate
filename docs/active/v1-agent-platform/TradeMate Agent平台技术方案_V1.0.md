# TradeMate Agent 平台技术方案

**文档版本**：V1.0  
**对应 PRD**：`docs/active/v1-agent-platform/TradeMate Agent平台产品需求文档_PRD_V1.0.md`  
**文档状态**：平台蓝图 + 首版落地版  
**生效日期**：2026-03-10  
**技术定位**：多 Agent 共用底座 + V1 广告 Agent 首版实现

## 1. 方案目标

### 1.1 技术目标

本方案同时解决两件事：

1. 建立 TradeMate 的多 Agent 共用底座。
2. 在该底座上交付 V1 广告 Agent。

### 1.2 架构原则

1. 平台先定义清楚，再收敛首版交付。
2. 首版按单 Agent 落地，架构按多 Agent 预留。
3. 插件为主入口，Web 为管理后台。
4. OpenClaw 作为底层能力层，不与核心业务层强耦合。

### 1.3 本期不做

1. 不做完整通用工作流平台。
2. 不做所有 Agent 一起落地。
3. 不做多平台统一执行。
4. 不做完全自治、无审批的自动化执行。

## 2. 平台级总体架构

### 2.1 总体分层

```text
TradeMate Agent Platform
├─ 入口层
│  ├─ Chrome 插件
│  └─ Web 管理后台
├─ 平台共用层
│  ├─ 账号与权限
│  ├─ 目标与约束中心
│  ├─ 情报与上下文中心
│  ├─ 建议中心
│  ├─ 审批中心
│  ├─ 任务中心
│  ├─ 通知中心
│  ├─ 审计中心
│  └─ 复盘与记忆中心
├─ Agent 能力层
│  ├─ 广告优化 Agent
│  ├─ 利润 Agent
│  ├─ 调价 Agent
│  ├─ 库存 Agent
│  └─ 合规 Agent
├─ 执行与运行时层
│  ├─ OpenClaw runtime / browser 能力
│  ├─ 动作执行器
│  ├─ 调度器
│  └─ 风险控制器
└─ 数据与基础设施层
   ├─ MySQL
   ├─ Redis
   ├─ 对象存储
   ├─ 日志监控
   └─ CI/CD
```

### 2.2 平台共用层职责

|模块|职责|
|---|---|
|目标与约束中心|统一管理 KPI、预算、风险偏好、阈值|
|情报与上下文中心|统一聚合站内数据、站外信号、用户历史行为|
|建议中心|统一结构化建议模型|
|审批中心|统一审批规则、审批流和处理记录|
|任务中心|统一任务状态机、执行队列、结果管理|
|通知中心|统一插件提醒、Web 消息和后续外部渠道|
|审计中心|统一记录所有关键动作|
|复盘与记忆中心|统一沉淀执行前后结果和用户偏好|

### 2.3 OpenClaw 在架构中的位置

OpenClaw 用于承接以下能力：

1. Agent runtime 和工具调用能力
2. 浏览器上下文和现场能力
3. 后续自动化执行扩展能力

OpenClaw 不直接承载：

1. 订阅与权限模型
2. 平台核心业务数据模型
3. 审批、审计、复盘等业务系统

## 3. 产品形态对应的技术职责

### 3.1 Chrome 插件

职责：

1. 展示多 Agent 待处理事项
2. 作为首版广告 Agent 的主交互入口
3. 承接即时提醒、单条审批、快速查看结果
4. 承接页面上下文与 OpenClaw 浏览器能力连接

### 3.2 Web 管理后台

职责：

1. 配置目标和约束
2. 批量审批和批量管理
3. 查看任务、审计、复盘
4. 后续承接多 Agent 的统一工作台

## 4. V1 实现范围：广告 Agent

### 4.1 为什么先实现广告 Agent

1. 动作清晰，易定义输入输出。
2. 反馈周期短，易验证效果。
3. 最容易验证 Agent 与执行闭环是否成立。
4. 最适合先打穿用户信任。

### 4.2 V1 实现边界

#### 平台底座实现

1. 账号与权限
2. 目标与约束配置
3. 建议模型
4. 审批服务
5. 任务中心
6. 通知服务
7. 审计服务
8. 复盘服务

#### 广告 Agent 实现

1. 广告目标配置
2. 广告建议生成
3. 风险分级
4. 审批触发
5. 任务执行
6. 结果回传
7. 复盘对比

## 6. 平台共用模型设计

### 6.1 共用实体

|实体|用途|
|---|---|
|user|用户账号|
|role|角色与审批权限|
|store|店铺|
|agent_goal|目标与约束|
|context_snapshot|上下文快照|
|suggestion|结构化建议|
|approval|审批单|
|task|执行任务|
|task_event|任务事件|
|notification|通知消息|
|audit_log|审计日志|
|review_snapshot|复盘快照|
|agent_memory|记忆摘要|

### 6.2 共用状态机

```text
suggestion_created
-> pending_approval
-> approved / rejected
-> task_queued
-> task_running
-> task_succeeded / task_failed / task_cancelled
-> review_generated
```

### 6.3 设计原则

1. 建议、审批、任务、复盘解耦。
2. 多 Agent 只扩展业务字段，不重造底层流程。
3. 所有关键对象显式记录 `agent_type`。

## 7. 广告 Agent 业务设计

### 7.1 广告 Agent 输入

1. 广告目标，如 ACOS、预算上限、风险偏好
2. 广告账户和活动表现数据
3. 搜索词、关键词、活动层级指标
4. 用户历史审批和执行结果

### 7.2 广告 Agent 输出

1. 结构化建议
2. 建议原因解释
3. 风险等级
4. 推荐动作
5. 预估影响

### 7.3 广告 Agent 正式动作

1. Campaign 预算调整
2. 关键词出价调整
3. Campaign 暂停
4. Campaign 恢复
5. 否词执行
6. 低效词降价或暂停
7. 高潜词加价

### 7.4 广告 Agent 实现策略

1. 首版以规则引擎为主。
2. LLM 负责建议解释、摘要和复盘说明。
3. 执行动作只接受结构化输出，不直接接受自然语言。

## 8. 插件主入口技术方案

### 8.1 插件组成

1. Popup 主窗口
2. Content Script
3. Service Worker
4. Options 设置页

### 8.2 插件交互职责

1. 展示待处理建议和风险等级
2. 处理单条批准、拒绝、稍后处理
3. 展示最近执行结果
4. 跳转 Web 进行复杂审批和复盘

### 8.3 通信方案

1. 插件业务请求通过 Service Worker 转发。
2. 插件与服务端使用 REST API + WebSocket。
3. Popup 与 Content Script 通过 `chrome.runtime` 消息机制同步。

## 10. 安全、审批与风控

### 10.1 权限链路

1. 登录态校验
2. 订阅与角色校验
3. Agent 动作权限校验
4. 审批状态校验

### 10.2 执行安全

1. 动作必须来自受控动作注册表。
2. 所有动作具备幂等键。
3. 高风险动作不允许绕过审批。
4. 连续失败达到阈值时自动熔断。

### 10.3 审计

1. 所有建议审批和任务执行必须写审计日志。
2. 审计需记录 agent 类型、目标对象、动作、结果和操作者。

3. 任务执行成功率
4. 审批平均耗时
5. 通知送达率
6. 审计覆盖率

### 11.2 V1 广告 Agent 指标

1. 广告建议生成成功率
2. 广告动作执行成功率
3. 复盘生成成功率
4. 插件待处理建议展示时延

## 12. 测试方案

### 12.1 平台底座测试

1. 建议到审批到任务到复盘的完整链路测试
2. 插件和 Web 状态一致性测试
3. 审计完整性测试

### 12.2 广告 Agent 测试

1. 建议生成规则准确性
2. 单条审批流程
3. 批量审批流程
4. 执行动作正确性
5. 复盘数据正确性

### 12.3 验收指标

1. 插件打开时间不超过 1 秒。
2. 建议首屏展示不超过 2 秒。
3. 任务状态更新不超过 10 秒。
4. 高风险任务审批覆盖率 100%。
5. 审计覆盖率 100%。

1. TradeMate 的产品总定义是多 Agent 平台。
2. V1 的正式交付是广告 Agent。
3. OpenClaw 作为底层能力层使用，不取代核心业务系统。
