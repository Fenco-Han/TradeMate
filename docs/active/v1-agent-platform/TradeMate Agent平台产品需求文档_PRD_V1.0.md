# TradeMate Agent 平台产品需求文档（PRD）

**文档版本**：V1.0  
**文档状态**：平台蓝图 + 首版落地版  
**生效日期**：2026-03-10  
**产品定位**：以 OpenClaw 能力为底座、以 Chrome 插件为主入口的跨境运营 Agent 平台  
**文档目标**：用于平台立项、版本规划、研发拆解、测试验收

## 1. 文档综述

### 1.1 编写目的

本文件用于同时定义两件事：

1. TradeMate 的长期产品蓝图：一个以多 Agent 协同为核心的跨境运营平台。
2. TradeMate V1 的正式交付：以广告 Agent 为切口，打通建议、审批、执行、复盘闭环。

### 1.2 背景判断

结合现有调研和市场变化，TradeMate 不应继续定义为“类似竞品的数据插件”，而应定义为“Agent-native 的运营系统”，原因如下：

1. 纯数据型插件已高度同质化，难以建立长期差异化。
2. OpenClaw 爆火后，市场对产品的关注点正在从“信息展示”转向“可执行能力”。
3. 卖家的主要矛盾不在于缺少更多面板，而在于缺少一套可控、可追踪、可持续优化的执行系统。
4. 插件形态依然重要，因为运营动作高度依赖浏览器现场上下文；但插件应是入口，而不是产品全部。

### 1.3 产品定义

TradeMate 定义为一个多 Agent 跨境运营平台：

1. 插件端负责现场识别、即时提醒、快捷处理和低摩擦入口。
2. Web 端负责策略配置、批量管理、审批、复盘和协作。
3. Agent 底座负责目标理解、建议生成、任务编排、执行回传和记忆沉淀。
4. OpenClaw 能力用于承接 Agent runtime、浏览器现场能力和后续执行扩展。

TradeMate 的核心产品逻辑不是“功能堆叠”，而是统一承接以下闭环：

`目标 -> 情报 -> 建议 -> 审批 -> 执行 -> 复盘 -> 记忆`

### 1.4 文档边界

本 PRD 覆盖：

1. TradeMate 的平台级能力蓝图。
2. 多 Agent 共享底座的产品结构。
3. V1 广告 Agent 的正式交付范围。

本 PRD 暂不细化：

1. 多平台正式扩展需求。
2. 低代码开放平台的详细交互。
3. 企业私有化部署的详细规格。

## 2. 平台蓝图

### 2.1 北极星定位

TradeMate 的长期目标不是一个单点 Agent，也不是一个功能更全的插件，而是一个以 Agent 为核心的跨境运营系统。

### 2.2 产品能力架构

TradeMate 平台能力分为六大能力域：

```text
TradeMate 平台能力架构
├─ 情报洞察中枢
│  ├─ 亚马逊站内数据抓取
│  ├─ 站外平台情报采集
│  ├─ 多源交叉验证
│  └─ 蓝海机会识别
├─ 多角色 AI Agent 决策层
│  ├─ 选品分析 Agent
│  ├─ 广告优化 Agent
│  ├─ 库存管理 Agent
│  ├─ 合规风控 Agent
│  └─ 经营分析 Agent
├─ 自动化执行引擎
│  ├─ 竞品监控与自动调价
│  ├─ 广告自动执行
│  ├─ Listing 自动优化
│  └─ 订单/库存自动化处理
├─ 合规风控体系
│  ├─ 规则订阅与变更解读
│  ├─ 全流程合规审计
│  └─ 风险预警与整改建议
├─ 现金流与利润管控
│  ├─ 实时利润核算
│  ├─ 现金流预测
│  └─ 成本优化与定价模拟
└─ 低代码开放平台
   ├─ 可视化工作流编排
   ├─ 技能市场
   └─ 开放 API 与企业集成
```

### 2.3 平台级共用底座

无论未来接入哪个 Agent，都应共享以下底座：

1. 统一账号与权限体系
2. 统一目标与约束配置
3. 统一情报与上下文模型
4. 统一建议模型
5. 统一审批流
6. 统一任务中心
7. 统一通知与审计
8. 统一复盘与记忆沉淀

### 2.4 版本策略

|阶段|定位|目标|
|---|---|---|
|V1|单 Agent 切口|用广告 Agent 验证平台闭环和用户信任|
|V2|双 Agent 扩展|接入利润 Agent 或调价 Agent，验证底座复用|
|V3|平台化成型|形成多 Agent 协同与统一工作流能力|

## 3. 产品形态

### 3.1 入口策略

TradeMate 采用“插件为主入口，Web 为管理后台”的形态：

|端|定位|职责|
|---|---|---|
|Chrome 插件|主入口|接收提醒、现场处理、快捷审批、快速查看状态|
|Web 后台|管理与复盘中心|配置目标、批量管理、审批、复盘、协作|
|Agent 底座|核心能力层|生成建议、调度任务、执行动作、沉淀记忆|

### 3.2 选择插件为主入口的原因

1. 卖家的主要工作现场就在浏览器内。
2. 插件更适合做低摩擦提醒和快速确认。
3. 插件更容易承接 OpenClaw 的浏览器上下文能力。
4. Web 端适合复杂分析，但不适合承接首个使用触点。

## 4. 用户与场景

### 4.1 目标用户

|用户类型|平台阶段价值|
|---|---|
|成长型卖家|V1 首要目标用户|
|广告专员|V1 核心高频用户|
|店铺负责人|V1 审批与复盘用户|
|规模化团队|V2-V3 重点扩展对象|

### 4.2 平台级核心场景

1. 用户在插件现场收到 Agent 建议并快速处理。
2. 用户在 Web 端查看多 Agent 待办、策略和复盘。
3. 系统按用户目标持续生成可执行动作。
4. 所有关键动作都经过审批、执行、审计和结果追踪。

## 5. V1 正式交付范围：广告 Agent

### 5.1 为什么从广告 Agent 切入

广告 Agent 是平台的第一个正式 Agent，而不是平台的全部定义。选择该切口的原因：

1. ROI 最容易量化。
2. 反馈周期短，能快速建立信任。
3. 建议和执行动作边界清晰。
4. 最适合验证“Agent 是否真的帮用户完成工作”。

### 5.2 V1 产品目标

1. 跑通广告 Agent 的建议、审批、执行、复盘闭环。
2. 让插件成为首要使用入口，而不是仅做数据展示。
3. 建立多 Agent 共用底座的第一版实现。

### 5.3 V1 功能范围

#### 5.3.1 平台底座能力

1. 账号与订阅体系
2. 目标与约束配置
3. 建议模型
4. 审批流
5. 任务中心
6. 通知与审计
7. 基础复盘

#### 5.3.2 广告 Agent 业务能力

1. 广告目标配置
2. 建议生成
3. 风险分级
4. 单条审批
5. 批量审批
6. 执行结果回传
7. 复盘分析

#### 5.3.3 V1 正式动作

1. Campaign 预算调整
2. 关键词竞价调整
3. Campaign 暂停
4. Campaign 恢复
5. 否词建议与执行
6. 低效词降价或暂停建议
7. 高潜词加价建议

### 5.4 V1 核心页面

#### 5.4.1 插件侧

1. 插件主窗口
2. 建议卡片列表
3. 单条审批操作
4. 最近执行结果
5. 高优先级提醒

#### 5.4.2 Web 侧

1. 广告 Agent 工作台
2. 待审批列表
3. 任务中心
4. 复盘分析页
5. 策略与目标配置页

### 5.5 V1 核心流程

#### 5.5.1 建议生成流程

1. 用户配置目标，如 ACOS、预算边界、风险偏好。
2. 系统聚合广告表现数据。
3. 广告 Agent 生成结构化建议。
4. 系统给出建议原因、风险等级和预估影响。

#### 5.5.2 审批执行流程

1. 用户在插件或 Web 查看建议。
2. 高风险建议进入审批流。
3. 审批通过后生成执行任务。
4. 系统执行动作并回传状态。

#### 5.5.3 复盘流程

1. 任务完成后生成执行前后对比。
2. 用户查看关键指标变化。
3. 结果进入 Agent 记忆上下文，用于后续建议。

### 5.6 V1 非目标范围

1. 不在 V1 做完整选品 Agent。
2. 不在 V1 做完整利润 Agent。
3. 不在 V1 做通用工作流编排器。
4. 不在 V1 做企业级开放生态。

## 6. 功能需求

### 6.1 账号与权限

1. 双端统一登录态。
2. 主账号、操作者、审批者角色区分。
3. 订阅权限控制 Agent 能力可见性和额度。

### 6.2 插件主窗口

1. 展示待处理建议数。
2. 展示高优先级广告异常。
3. 支持单条批准、拒绝、稍后处理。
4. 支持跳转 Web 查看详情和批量处理。

### 6.3 广告 Agent 工作台

1. 展示当前目标与约束。
2. 展示建议来源和推荐原因。
3. 支持重新生成建议。
4. 支持筛选高风险、低风险、待审批建议。

### 6.4 建议与任务中心

1. 查看建议草稿。
2. 发起任务。
3. 查看任务状态。
4. 查看执行结果和失败原因。

### 6.5 审批中心

1. 单条审批。
2. 批量审批。
3. 审批备注。
4. 审批历史查询。

### 6.6 复盘分析

1. 查看动作前后广告指标变化。
2. 查看建议采纳率、执行成功率、审批耗时。
3. 查看目标达成情况。

### 6.7 通知与审计

1. 插件提醒。
2. Web 消息中心。
3. 审计日志不可删除。
4. 所有关键动作可追溯。

## 7. 非功能需求

### 7.1 性能

1. 插件主窗口打开时间不超过 1 秒。
2. 建议首屏展示时间不超过 2 秒。
3. 任务状态更新到前端不超过 10 秒。

### 7.2 稳定性

1. 核心任务链路月可用性不低于 99.9%。
2. 高风险动作不得绕过审批。
3. 插件异常不影响亚马逊页面正常使用。

### 7.3 安全与合规

1. 执行链路全量审计。
2. 账号、权限、审批、动作四层校验。
3. 插件权限遵循最小必要原则。

## 8. 验收标准

### 8.1 平台底座验收

1. 建议、审批、任务、通知、审计可完整串联。
2. 插件和 Web 状态保持一致。
3. 单 Agent 不需要重新实现一套底座能力。

### 8.2 V1 广告 Agent 验收

1. 用户可在插件中查看并处理建议。
2. 用户可在 Web 侧批量审批。
3. 系统可执行至少 7 类广告相关动作或建议执行。
4. 每条任务均有审计记录和结果回传。
5. 用户可查看复盘数据。

## 9. 路线图

### 9.1 V1

广告 Agent 首版闭环。

### 9.2 V2

在复用底座的前提下，引入利润 Agent 或调价 Agent。

### 9.3 V3

逐步补齐情报、库存、合规、低代码平台等能力域。

## 10. 默认假设

1. 多 Agent 平台是产品总定义。
2. 广告 Agent 是 V1 首个正式交付 Agent。
3. 插件是主入口，不改成 Web-first。
4. 文档命名和产品表达以 Agent 平台为中心，而不是以插件为中心。
